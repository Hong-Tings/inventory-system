## 4. 开发规范

### 4.1 Git 分支策略

```
main        — 生产分支，仅合并 release
├─ dev      — 开发主分支
│  ├─ feat/product      — 商品模块
│  ├─ feat/purchase     — 入库模块
│  ├─ feat/sales        — 出库模块
│  ├─ feat/inventory    — 库存模块
│  └─ feat/stocktake    — 盘点模块
└─ release/v1.0         — 发布候选
```

- 功能分支从 `dev` 拉出，完成后 PR → `dev`
- 阶段交付时 `dev` → `release/v1.0` → 测试 → `main`

### 4.2 代码规范

- Java：遵循 Alibaba Java Coding Guidelines
- Vue3：使用 Composition API + `<script setup>`
- UniApp：遵循 Vue3 组合式 API
- API 路径小写 + 中划线：`/api/v1/purchase-order`
- 数据库表名小写下划线：`purchase_order`
- 提交信息格式：`[模块] 描述`，如 `[purchase] 实现入库单提交事务`

### 4.3 前后端约定

- 所有接口返回统一格式 `Result<T>`
- 分页参数统一：`page, size, sort, order`
- 时间传输使用 ISO 8601 字符串
- 金额使用 Decimal 类型（BigDecimal），精度 2 位

## 5. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 需求频繁变更 | 中 | 高 | 固定 M1 需求基线，变更走 CR 流程 |
| 人员变动 | 低 | 高 | 关键模块至少两人熟悉 |
| 小程序审核周期长 | 中 | 中 | 提前准备审核材料，预留 7 天审核期 |

| 第三方依赖（微信登录）变更 | 低 | 中 | 关注官方公告，及时适配 |

## 6. 验收标准

| 验收项 | 标准 |
|--------|------|
| 采购入库 | 正常创建→提交→库存+1，取消→库存回滚 |
| 销售出库 | 正常创建→提交→库存-1，超库存无法提交 |
| 库存盘点 | 全盘/抽盘→录入→审核→差异调整，库存正确 |
| 库存查询 | 按仓库/商品/库位多维度查询结果正确 |
| 库存流水 | 每笔出入库可追溯 |
| 报表 | 出入库统计数据与实际单据一致 |
| 权限 | 不同角色看到不同菜单和操作按钮 |
| 并发 | 50 并发下单不超卖、不崩溃 |
