## 二、审批工作流状态机

```
DRAFT(0) ──submit──→ PENDING(4) ──approve──→ CONFIRMED(1)
   ↑                       │
   └───────reject──────────┘
   │
   ├──updateDraft → 编辑（仅草稿）
   ├──cancel → CANCELED(2)
   
PENDING(4) ──cancel(仅管理员)──→ CANCELED(2)
PENDING(4) ──void(仅管理员)────→ VOIDED(3)
CONFIRMED(1) ──cancel(回滚库存)──→ CANCELED(2)
```

## 三、权限控制

| 审核通过 (PENDING→CONFIRMED) | ❌ | ✅ (@SaCheckRole) |
| 驳回 (PENDING→DRAFT) | ❌ | ✅ (@SaCheckRole) |
| 作废 | ❌ | ✅ (@SaCheckRole) |
| 删除 | ❌ | ✅ (@SaCheckRole) |

## 四、库存处理

| 单据类型 | 审核通过 | 取消/回滚 |
|---------|---------|----------|
| 采购入库 | 增加库存 + 移动加权平均成本 | 扣减库存 + 重算成本 |
| 销售出库 | FIFO 扣减库存 | 恢复库存 |
| 调拨 | 调出仓扣减 + 调入仓增加 + 成本转移 | 双向回滚 |

## 五、审核审计

- `approve()` 自动记录 `approver_id`（当前登录用户ID）和 `approve_time`
- 查询时通过 `enrichOrder()` 加载 `approverName`（审核人姓名）
- 驳回原因记录在 `remark` 字段（追加 `"驳回原因: xxx"`）

## 七、已知的非标准设计

1. **驳回无独立状态** — 驳回后回到 `DRAFT(0)`，与新建草稿状态相同。驳回原因通过 remark 字段记录。符合项目实际需求。
2. **盘点独立流程** — 盘点单使用独立的 `0→1→2` 状态流转（盘点中→已审核→已调整），不使用标准审批流。

# 仓库分级改造记录

## 一、设计概述

仓库支持 1~4 级父子层级结构，使用自引用树（`parent_id` + `level`）实现：

- **1 级**：最大层级（如"华东仓储中心"）
- **2 级**：子区域（如"上海分仓"）
- **3 级**：子区域（如"浦东新区仓"）
- **4 级**：最细粒度（如"浦东常温区"）

**规则：** 只有叶子节点（无子仓库的节点）存储实际库存，父级节点自动汇总子级库存。

## 三、仓库树结构

```
华东仓储中心 (1级)
├── 上海分仓 (2级)
│   ├── 浦东新区仓 (3级)
│   │   ├── 浦东常温区 (4级) ← 实际存储库存
│   │   └── 浦东冷链区 (4级) ← 实际存储库存
│   └── 松江仓 (3级) ← 叶子节点，实际存储库存
└── 杭州分仓 (2级)
    └── 萧山仓 (3级) ← 叶子节点，实际存储库存
```

- 叶子节点判定：`hasChildren(id)` 实时查询，不依赖 level 硬编码
- 库存汇总：父级查询时自动 `SUM` 所有子级库存

## 四、关键业务规则

| 规则 | 实现 |
|------|------|
| 等级必须连续 | `save()` 校验 `parent.getLevel() + 1 == newChild.getLevel()` |
| 有库存不可新增子级 | `update()` 检查 `inventoryMapper.countByWarehouseId()` |
| 有子仓库不可删除 | `delete()` 检查 `warehouseMapper.countByParentId()` |
| 有库存不可更换父级 | `update()` 阻止修改 `parent_id` |
| 父级不可选择存储 | `listAll()` 只返回无子节点的仓库 |
